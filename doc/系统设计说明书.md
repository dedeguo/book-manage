# 校园图书管理系统——系统设计说明书

> 学生简版  
> 本文档说明系统准备“怎样实现”。它承接《需求分析说明书》，重点介绍系统结构、功能模块、数据库、接口和用户界面，不要求学生在设计阶段写出全部代码。

## 1. 设计目标

校园图书管理系统采用一个结构清楚、容易运行和调试的 Java Web 单体项目。设计时主要考虑：

- 普通用户与管理员权限分开。
- 图书、用户和借阅记录的数据关系清楚。
- 借书、还书形成完整业务闭环。
- 页面操作简单，错误提示明确。
- 各层职责清楚，便于学生理解和分工。
- 首先保证核心功能正确，再考虑界面优化和扩展功能。

本项目不使用复杂的微服务、消息队列和分布式架构。对于课程项目，能理解、能运行、能测试比堆砌技术更重要。

## 2. 系统总体设计

### 2.1 系统架构

系统采用 B/S 架构。用户通过浏览器访问系统，Tomcat 运行 Java Web 应用，MySQL 保存用户、图书和借阅数据。

~~~mermaid
flowchart TB
    A["浏览器页面 HTML / CSS / JavaScript"] --> B["权限过滤器 Filter"]
    B --> C["请求控制 Servlet"]
    C --> D["业务处理 Service"]
    D --> E["数据访问 DAO"]
    E --> F[("MySQL 数据库")]
~~~

一次请求的基本过程：

1. 页面通过 HTTP 请求访问系统接口。
2. Filter 检查用户是否登录、是否具有相应权限。
3. Servlet 接收参数并调用 Service。
4. Service 执行业务规则，例如借阅上限、超期检查和库存更新。
5. DAO 负责查询或修改数据库。
6. 处理结果以 JSON 返回页面，页面更新显示内容。

### 2.2 分层职责

| 层次 | 主要职责 | 项目中的示例 |
|---|---|---|
| 页面层 | 显示数据、收集输入、调用接口、反馈结果 | user、admin 目录中的 HTML 页面 |
| 过滤器层 | 检查登录状态和角色权限 | LoginFilter、AdminFilter |
| 控制层 | 接收请求、读取参数、返回 JSON | AuthServlet、BookServlet、BorrowServlet |
| 业务层 | 执行核心业务规则 | UserService、BookService、BorrowService |
| 数据访问层 | 对数据库进行增删改查 | UserDao、BookDao、BorrowRecordDao |
| 数据层 | 保存用户、图书和借阅记录 | MySQL 数据库 |

学生可以把它简单理解为：页面负责“展示”，Servlet 负责“接收请求”，Service 负责“判断怎么办”，DAO 负责“操作数据库”。

## 3. 功能模块设计

### 3.1 模块与代码对应关系

| 功能模块 | 主要内容 | 主要控制器 |
|---|---|---|
| 用户认证 | 注册、登录、退出 | AuthServlet |
| 个人中心 | 查看资料、修改邮箱和密码 | UserServlet |
| 图书查询 | 图书列表、搜索、分类和详情 | BookServlet |
| 我的借阅 | 借书、还书、当前和历史记录 | BorrowServlet |
| 管理员图书管理 | 图书新增、修改、删除和查询 | AdminBookServlet |
| 管理员用户管理 | 用户查询和删除 | AdminUserServlet |
| 管理员借阅管理 | 全部记录、超期记录、代还 | AdminBorrowServlet |

### 3.2 用户认证模块

用户注册时，系统检查学号或工号是否重复，并根据账号格式判断学生或教师身份。密码经过散列处理后保存，数据库中不能保存明文密码。

用户登录成功后，服务器在 Session 中保存用户身份。普通用户进入用户首页，管理员进入管理后台。退出登录时清除 Session。

### 3.3 图书查询模块

图书模块负责：

- 查询全部图书。
- 按书名、作者或 ISBN 模糊搜索。
- 按分类筛选。
- 查询图书详情。
- 返回总复本数和可借复本数。

图书是否可以借阅不单独保存为文字状态，而是根据 available_copies 是否大于 0 动态判断。

### 3.4 借阅管理模块

借阅模块是系统最重要的业务模块。

借书时需要依次检查：

1. 用户当前借阅数量是否达到 5 本。
2. 用户是否存在超期未还图书。
3. 图书可借复本数是否大于 0。
4. 用户是否重复提交同一次借阅操作。

全部检查通过后，系统创建借阅记录，应还日期为借阅日期后 30 天，同时将图书可借复本数减 1。

还书时，系统更新借阅状态和实际归还日期，并将可借复本数加 1。借阅记录和图书数量的修改应作为一次完整业务处理，避免只修改其中一项。

### 3.5 借阅时序设计

~~~mermaid
sequenceDiagram
    actor U as 用户
    participant P as 用户页面
    participant C as BorrowServlet
    participant S as BorrowService
    participant D as DAO

    U->>P: 点击借阅
    P->>C: 提交图书ID
    C->>S: borrow(userId, bookId)
    S->>D: 查询借阅数、超期记录和库存
    D-->>S: 返回查询结果
    alt 不满足借阅条件
        S-->>C: 返回失败原因
        C-->>P: 显示错误提示
    else 满足借阅条件
        S->>D: 新增借阅记录并减少可借复本
        D-->>S: 保存成功
        S-->>C: 借阅成功
        C-->>P: 更新页面
    end
~~~

### 3.6 管理员模块

管理员端负责维护系统数据：

- 图书管理：新增、修改、删除和搜索图书。
- 用户管理：查询用户，删除符合条件的普通用户。
- 借阅管理：查询全部借阅记录、超期记录和代为归还。
- 统计概览：显示图书、用户、借阅和超期数量。

删除操作必须先检查业务条件。例如，仍有复本借出的图书不能删除，仍有图书未归还的用户不能删除，管理员账号不能删除。

## 4. 数据库设计

### 4.1 数据关系

系统包含用户、图书和借阅记录三个核心实体。用户与图书之间是多对多关系，通过借阅记录建立联系。

~~~mermaid
erDiagram
    USER ||--o{ BORROW_RECORD : "产生"
    BOOK ||--o{ BORROW_RECORD : "对应"

    USER {
        int id
        string username
        string name
        string role
        string user_type
    }

    BOOK {
        int id
        string title
        string isbn
        int total_copies
        int available_copies
    }

    BORROW_RECORD {
        int id
        int user_id
        int book_id
        datetime borrow_date
        datetime due_date
        datetime return_date
        string status
    }
~~~

### 4.2 用户表 users

| 字段 | 说明 |
|---|---|
| id | 用户主键 |
| username | 学号或工号，不能重复 |
| name | 用户姓名 |
| password | 散列后的密码 |
| email | 邮箱 |
| role | user 或 admin |
| user_type | student、teacher 或 admin |
| created_at | 注册时间 |

### 4.3 图书表 books

| 字段 | 说明 |
|---|---|
| id | 图书主键 |
| title | 书名 |
| author | 作者 |
| publisher | 出版社 |
| isbn | ISBN |
| description | 图书简介 |
| cover_url | 封面图片地址 |
| category | 图书分类 |
| total_copies | 总复本数 |
| available_copies | 当前可借复本数 |
| created_at | 添加时间 |

### 4.4 借阅记录表 borrow_records

| 字段 | 说明 |
|---|---|
| id | 借阅记录主键 |
| user_id | 借阅用户 |
| book_id | 被借图书 |
| borrow_date | 借阅日期 |
| due_date | 应还日期 |
| return_date | 实际归还日期，未归还时为空 |
| status | borrowing 或 returned |

### 4.5 数据一致性规则

- available_copies 不能小于 0，也不能大于 total_copies。
- 借阅成功时，新增借阅记录并减少可借复本数。
- 归还成功时，更新借阅记录并增加可借复本数。
- 已归还的借阅记录不能再次归还。
- 删除用户和图书前必须检查是否存在未完成借阅。
- 用户、图书和借阅记录通过主键建立关联，不能保存不存在的用户或图书编号。

## 5. 接口设计

页面通过统一的 API 与后端交互，接口返回 JSON 数据。

| 模块 | 接口路径 | 主要功能 |
|---|---|---|
| 用户认证 | /api/auth | 注册、登录、退出 |
| 个人中心 | /api/user | 查询资料、修改密码和邮箱 |
| 图书查询 | /api/books | 图书列表、搜索、分类和详情 |
| 用户借阅 | /api/borrow | 借阅、归还、当前和历史记录 |
| 管理员图书 | /api/admin/books | 图书增删改查 |
| 管理员用户 | /api/admin/users | 用户查询和删除 |
| 管理员借阅 | /api/admin/borrows | 全部记录、超期记录和代还 |

统一返回格式：

~~~json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
~~~

页面不能只根据 HTTP 请求是否发送成功判断结果，还需要读取 code 和 message，并把错误原因显示给用户。

### 6.2 整体界面风格

系统分为用户端和管理端两套界面。

| 界面 | 设计风格 |
|---|---|
| 用户端 | 顶部导航，紫色渐变作为主色，卡片式内容布局 |
| 管理端 | 左侧深色菜单，右侧内容区，以表格和操作栏为主 |
| 页面背景 | 浅灰色，主要内容放在白色卡片中 |
| 状态颜色 | 绿色表示正常或可借，红色表示超期或危险操作，灰色表示不可用 |
| 反馈方式 | 成功提示、错误提示、空数据提示、加载状态和二次确认 |

页面设计应优先保证信息清楚，不要求学生追求复杂动画。

界面要求：

- 输入框必须有清楚的标签和提示。
- 密码可以切换显示或隐藏。
- 登录按钮应突出显示。
- 账号或密码错误时，在表单附近显示明确提示。

- 应还日期。
- 实际归还日期。
- 借阅状态。

### 6.13 通用交互要求

- 页面加载数据时显示加载状态，避免用户重复点击。
- 表单提交前检查必填项和数据格式。
- 新增、修改和删除后及时刷新列表。
- 删除操作必须二次确认。
- 空列表显示“暂无数据”，不能只留下空白页面。
- 操作失败时显示具体原因，不只显示“系统错误”。
- 按钮在不可操作时使用禁用状态。
- 表格在小屏幕下允许横向滚动。
- 用户端与管理端同类按钮、颜色和提示方式保持一致。

## 7. 权限与异常处理设计

### 7.1 权限控制

| 功能 | 未登录用户 | 普通用户 | 管理员 |
|---|---:|---:|---:|
| 注册、登录 | 允许 | 允许 | 允许 |
| 图书查询 | 禁止 | 允许 | 允许 |
| 借阅和归还 | 禁止 | 允许本人操作 | 可代还 |
| 个人资料 | 禁止 | 只能查看和修改本人 | 查看本人 |
| 图书管理 | 禁止 | 禁止 | 允许 |
| 用户管理 | 禁止 | 禁止 | 允许 |
| 全部借阅记录 | 禁止 | 禁止 | 允许 |

LoginFilter 负责检查是否登录，AdminFilter 负责检查是否为管理员。前端隐藏按钮只能改善界面，真正的权限检查必须放在服务端。

### 7.2 异常处理

| 异常情况 | 系统处理 |
|---|---|
| 未登录或 Session 失效 | 返回登录页面或提示重新登录 |
| 普通用户访问管理接口 | 拒绝访问并返回无权限提示 |
| 参数为空或格式错误 | 指出具体字段问题 |
| 图书不存在 | 提示图书不存在或已被删除 |
| 图书无可借复本 | 拒绝借阅并提示暂无库存 |
| 达到借阅上限 | 提示最多同时借阅 5 本 |
| 存在超期未还 | 提示先归还超期图书 |
| 重复归还 | 拒绝操作，不重复增加库存 |
| 数据库操作失败 | 回滚本次操作并记录错误日志 |

## 8. 部署设计

系统采用简单的单机部署，适合课程实验和课堂演示。

    C --> D[("MySQL 8")]

部署要求：

- 安装 JDK 17、Tomcat 11、MySQL 8 和 Maven。
- 创建 library_db 数据库并执行 doc/sql/book.sql。
- 修改 Druid 数据库连接配置。
- 使用 Maven 打包生成 book.war。
- 将 WAR 文件部署到 Tomcat 的 webapps 目录。
- 系统访问路径使用 /book。

## 9. 建议开发顺序

为了降低难度，建议按以下顺序实现：

1. 创建数据库和三张核心表。
2. 完成项目分层结构和数据库连接。
3. 完成注册、登录、退出和权限过滤。
4. 完成图书列表、搜索、分类和详情。
5. 完成借书、还书和我的借阅。
6. 完成管理员图书、用户和借阅管理。
7. 对照界面设计统一页面风格。
8. 测试借阅上限、无库存、超期、重复提交和权限控制。
9. 打包并部署到 Tomcat。

每完成一个模块，都应先独立测试，再继续开发后续模块，不要等所有代码写完后才统一测试。

## 10. 设计总结

本系统采用浏览器、Servlet、Service、DAO 和 MySQL 组成的分层结构。设计的核心不是使用复杂技术，而是保证：

- 用户和管理员职责清楚。
- 图书数量与借阅记录始终一致。
- 借书和还书流程可以完整运行。
- 权限和异常情况得到处理。
- 用户端和管理端页面清楚、统一、容易操作。

学生在编码过程中可以调整类名和页面细节，但不能随意改变需求中的核心业务规则。如果设计发生较大变化，应先更新本文档，再修改代码。
