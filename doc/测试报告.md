## 4. Postman API 接口测试

### 4.1 Postman 基本设置

在 Postman 中创建 Collection：`校园图书管理系统`，再创建环境变量：

| 变量名 | 初始值 |
|---|---|
| `baseUrl` | `http://localhost:8080/book` |
| `bookId` | 选择一本可借图书的 ID |
| `recordId` | 选择一条借阅中记录的 ID |

POST 请求统一设置：

- Header：`Content-Type: application/json`
- Body：选择 `raw` 和 `JSON`
- 请求地址写成 `{{baseUrl}}/api/...`

系统使用 Session 保存登录状态。Postman 通常会自动保存 `JSESSIONID` Cookie，因此必须先执行登录接口，再测试需要登录的接口。切换普通用户和管理员时，应先退出或清除旧 Cookie。

接口返回 JSON，例如：

~~~json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
~~~

本项目部分业务错误可能仍返回 HTTP 200，因此判断是否成功时应重点检查响应体中的 `code`、`message` 和 `data`，不能只看 HTTP 状态码。

可在 Postman 的 Tests 中加入简单断言：

~~~javascript
pm.test("返回内容是 JSON", function () {
    pm.response.to.be.json;
});

pm.test("业务处理成功", function () {
    const result = pm.response.json();
    pm.expect(result.code).to.eql(200);
});
~~~

### 5.2 界面检查要点

测试每个页面时顺便检查：

- 页面标题、按钮和表单标签是否清楚。
- 图书封面加载失败时是否影响主要操作。
- 表单必填项为空时是否有提示。
- 删除操作是否要求再次确认。
- 操作成功或失败后是否显示明确消息。
- 列表没有数据时是否显示“暂无数据”。
- 当前菜单是否高亮，页面跳转是否正确。
- 浏览器刷新后登录状态是否合理。
- 表格较宽时是否可以正常查看。
